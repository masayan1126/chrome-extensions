# YouTube台本 改善計画: 実際のセッションやり取りを追加する

## Context

**対象ファイル:** `/Users/masayan/git/content/youtube/deep-tech/chrome-extension-agentic-coding/output/anthropic-workflow-script.md`

**現状の問題:**
台本は構成・リズム・強調マーカーが整っているが、「実際のセッションでClaudeとユーザーがどんなやり取りをしたか」が抽象的で、視聴者が「本当にこう動くのか」を確認できない。ツール名と結果だけが書かれており、プロセスが見えない。

**目標:**
各フェーズに「ユーザーが入力したこと → Claude が返したこと」という具体的なやり取りを追加し、視聴者がセッションを疑似体験できる台本にする。

---

## 改善が必要な箇所（5箇所）と改善内容

### 改善①：ガードレール（2:30〜4:00）— security-guidance のフック発火を実物で見せる

**現状:** 「XSSやインジェクション系のパターンを検知すると〜ファイルの書き込み自体をブロックしてくれます」と説明するのみ

**改善内容:** 実際にブログ記事を書いている最中にフックが2回発火したエピソードを追加する。これは「動画の制作中にも起きた」という事実なので信頼性が最も高い。

追加するナレーション例：
```
実は今回、このブログ記事を書いているときに2回フックが発火しました。
1回目は、セキュリティ警告の説明文の中に特定のキーワードを書いたとき。
2回目は、ReactのXSS対策に関する解説文の中のキーワードを書いたとき。
ドキュメントを書くだけでも止まる。それくらい厳密に動いています。
```

---

### 改善②：フェーズ①（4:00〜5:30）— doc-coauthoring の対話内容を見せる

**現状:** 「Claude Code が対話形式で仕様書を一緒に作ってくれます。今回は5つのセクションを詰めました。」の1文だけ

**改善内容:** 実際のやり取りの流れを短く再現する。視聴者は「どんな質問がくるのか」を最も知りたがっている。

追加するナレーション例：
```
Claude Code はまずこんな質問を投げてきます。
「解決したい問題を具体的に教えてください」
そこで自分はこう答えます。
「異なるフォルダにある同名ファイルを同時にタブで開けない」
すると次に——
「影響するコンポーネントはどのあたりですか? タブ管理・サイドバー・D&Dのどれですか?」
こうやって5つのセクション分、一問一答で詰めていくイメージです。
```

---

### 改善③：フェーズ②（5:30〜7:00）— feature-dev の設計選択シーンを見せる

**現状:** 「code-architect が提案したアーキテクチャは isSameEntry() API でのファイル同一性判定」と結論だけを説明

**改善内容:** code-architect が複数の選択肢を提示し、ユーザーが選んだという事実を再現する。実際のセッションでは「B」を選んだ。

追加するナレーション例：
```
code-architect が2つのアプローチを提案してきました。
Aは「FileInfo 型にユニークIDフィールドを追加する」
Bは「FileSystemFileHandle.isSameEntry() でファイルシステムレベルで判定する」
自分はBを選びました。
型に余計なフィールドを増やさない方が、後で不整合が起きにくいからです。
この選択がレビューフェーズで評価されることになります。
```

---

### 改善④：フェーズ⑥（14:30〜17:00）— pr-review-toolkit の並列起動の様子を見せる

**現状:** 「code-reviewer・type-design-analyzer・silent-failure-hunter が並列で動きます」と説明するのみ

**改善内容:** 実際に起動したときの出力の雰囲気を読み上げ形式で追加する。視聴者は「並列でどう見えるのか」を知りたい。

追加するナレーション例：
```
pr-review-toolkit を呼び出すと、こんな感じで3つのエージェントが同時に走り出します。
「code-reviewer: PR の差分を確認中...」
「type-design-analyzer: 型定義を解析中...」
「silent-failure-hunter: エラーハンドリングをスキャン中...」
人間がレビューするなら順番に見ていくところを、同時並行でやってくれる。
```

---

### 改善⑤：フェーズ⑦（17:00〜18:30）— internal-comms への入力と2種類の出力を対比させる

**現状:** 「2種類が一度に生成されます」で終わっている

**改善内容:** 実際に生成されたリリースノートの冒頭を読み上げ、技術者向けとユーザー向けの文体の違いを声で対比して見せる。

追加するナレーション例：
```
技術者向けはこんな文体です。
「FileSystemFileHandle.isSameEntry() API を使った正確なファイル同一性判定に変更。
directoryReader にてフルパスを正しく構築するよう修正——」
（間）
一方、Chromeストア向けはこうなります。
「フォルダが違えば同じ名前のファイルでも、それぞれ別タブとして開けるようになりました。」
同じ内容を、読む人に合わせて自動で書き分けてくれる。これが internal-comms の価値です。
```

## 検証基準

- 各フェーズに「ユーザーが入力したこと」または「Claudeが返したこと」が最低1行含まれること
- security-guidance フック発火のエピソードが具体的に言及されること
- 設計フェーズで「選択した」という能動的な行動が表現されること

## 題材の背景（Issue #10）

- **問題:** `dir1/README.md` と `dir2/README.md` を同時にタブで開こうとすると片方しか開けない
- **根本原因:** `useTabOpen.ts` が `file.path` のみで重複チェックしていたため、フルパスが取得できていなかった
- **修正:** `FileSystemFileHandle.isSameEntry()` で正確なファイル同一性判定を導入
- **UI改善:** 同名ファイルが複数タブにある場合のみ `dir/filename` 形式でラベルを表示
